### Wiki 生成要求

**重要**：请首先深入分析代码库的实际内容、架构特点、目标用户群体和使用场景，然后根据项目特性**动态调整**文档结构。以下为可选的文档分类模块，请根据项目实际情况选择、合并、拆分或创建新的分类。

#### 核心模块（必选）
- **项目概述**
  - 项目简介（3-5句话）
  - 核心功能列表
  - 设计理念/架构原则

#### 可选模块（按需选择）
- **快速开始**
  - 前置条件（环境、依赖）
  - 安装步骤
  - 基本使用示例

- **架构说明**
  - 整体架构图（使用 Mermaid 或 ASCII）
  - 核心模块说明（每个模块的职责）
  - 模块间关系/数据流

- **API 参考**（库/API项目必选）
  - 主要接口/类列表
  - 关键参数说明
  - 使用示例代码

- **配置说明**
  - 配置文件位置
  - 可配置项列表
  - 每个配置项的说明和默认值

- **开发指南**
  - 项目结构说明
  - 代码规范（风格、命名约定）
  - 如何添加新功能
  - 测试要求

- **部署说明**（应用项目必选）
  - 构建流程
  - 部署环境要求
  - 常见部署方式
  - 环境变量说明

- **故障排查**
  - 常见问题及解决方案
  - 调试技巧
  - 日志位置和分析方法

- **贡献指南**
  - 如何提交代码
  - Pull Request 流程
  - 代码审查标准

- **更新日志**
  - 最近的重大变更
  - 版本说明

#### 文档结构化策略

根据项目特性，采用以下策略之一：

1. **功能导向**：按功能模块组织（适合多功能应用）
2. **用户角色**：按使用者分类（如：开发者、运维、最终用户）
3. **使用场景**：按使用流程组织（适合工具类项目）
4. **技术层次**：按技术深度组织（适合框架/库项目）

**要求**：在生成文档前，先说明你选择的结构策略及理由。

### 文档格式要求

- 使用 Markdown 格式
- 代码块使用语法高亮
- 重要概念使用粗体
- 列表项使用无序列表
- 适当使用表格展示配置项和参数

### 输出方式

根据项目规模和复杂度，选择以下输出方式之一：

#### 方式一：单文件输出（适合小型项目）

- 将所有内容合并为一个 Markdown 文件
- 文件内部使用清晰的章节导航（目录）
- 适合：简单工具、小型库、概念验证项目

#### 方式二：多文件输出（适合中大型项目）

- **主文件（Home）**：目录导航，包含：
  - 项目简介
  - 文档结构总览
  - 各子文档的链接和简要说明
  - 快速导航指南

- **子文档**：按模块/功能/角色拆分为独立文件
  - 每个文件聚焦一个主题
  - 文件间通过链接互相引用
  - 适当使用 `[[文件名]]` 或相对路径链接

- 适合：复杂应用、框架、多模块项目、面向多用户群体的项目

**文件命名建议**：
- 主文件：`home.md`
- 子文件：使用清晰的命名，如：
  - `installation.md`（安装指南）
  - `architecture.md`（架构说明）
  - `api-reference.md`（API 参考）
  - `deployment.md`（部署说明）
  - `developer-guide.md`（开发者指南）

**选择依据**：
- 项目代码行数 < 5000：建议单文件
- 文档总长度 > 20 页：必须多文件
- 有多个不同用户群体：建议多文件
- 需要频繁更新某些模块：建议多文件

### 文档结构化原则

**灵活性优先**：
- 没有固定的文档结构，完全基于项目实际内容
- 可以合并、拆分、重命名模块以适应项目特点
- 可以创建新的分类来更好地组织内容

**以用户为中心**：
- 考虑目标用户的阅读场景和信息需求
- 将用户最需要的信息放在最前面
- 提供清晰的导航和索引

**实用性与完整性平衡**：
- 确保覆盖所有重要的使用场景
- 避免冗余和无关内容
- 提供具体可操作的指导
